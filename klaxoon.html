1940s – 1950s

Assembly &
Machine Code

You speak the machine's language — raw instructions, memory addresses, CPU registers. Every line is a direct command to the hardware.

"The programmer is the translator between human intent and electrical impulse. There is no abstraction. Only voltage."

_start:
  mov  eax, 4    ; sys_write

1972 – 1990s

C & Structured
Programming

Power meets portability. Write once, compile anywhere. C gives you control over memory while freeing you from assembly syntax. Unix is written in it. So is everything after.

"C is the bedrock. Most modern languages compile down to it, or run on a runtime written in it. You're always one layer above C."

sort.c
C · 1975
/* Quicksort — elegant, dangerous, fast */
/* You manage memory. You own the pointers. */

#include <stdio.h>

void swap(int *a, int *b) {
  int tmp = *a;
  *a = *b;
  *b = tmp;
}

void quicksort(int arr[], int lo, int hi) {
  if (lo < hi) {
    int pivot = arr[hi];
    int i = lo - 1;
    for (int j = lo; j < hi; j++) {
      if (arr[j] <= pivot)
        swap(&arr[++i], &arr[j]);
    }
    swap(&arr[i + 1], &arr[hi]);
    int p = i + 1;
    quicksort(arr, lo, p - 1);
    quicksort(arr, p + 1, hi);
  }
}

1980s – 2000s

Object-Oriented
Programming

Model the world as objects. Encapsulate data and behavior together. Inheritance, polymorphism, abstraction — think in concepts, not bytes.

"OOP is the first time we asked code to resemble how humans think about the world — in things, categories, and relationships."

Account.java
Java · 1995
// Real-world concepts become code objects

public class BankAccount {
  private String owner;
  private double balance;

  public BankAccount(String owner) {
    this.owner = owner;
    this.balance = 0.0;
  }

  public void deposit(double amount) {
    if (amount > 0) balance += amount;
  }

  public boolean withdraw(double amount) {
    if (amount > balance) return false;
    balance -= amount;
    return true;
  }
}

The fundamental change

The end of
determinism

Every programming language ever created shared one property: the same input always produced the same output. This was the contract. LLMs broke it — and that changes everything.

⚙️
Traditional Code
Deterministic
Input: sort([3, 1, 2])
Run 1
[1, 2, 3]
Run 2
[1, 2, 3]
Run 3
[1, 2, 3]
Run 4
[1, 2, 3]
Identical input → identical output.
Every single time. Guaranteed by mathematics.
🧠
LLM (Natural Language)
Non-Deterministic
Same prompt, 4 times
"Write a function to validate an email address"
Run 1
const isValid = (e) => /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(e)
Run 2
function validateEmail(email) { return email.includes('@') && email.includes('.') && email.length > 5; }
Run 3
import validator from 'email-validator'; export const check = (e) => validator.validate(e);
Run 4
const EMAIL_RE = /^[\w.-]+@[\w.-]+\.[a-z]{2,}$/i; const validate = (e) => EMAIL_RE.test(e.trim());
Different every run · Cycling live ↑
🎯
You can't unit test a prompt
Traditional TDD assumes determinism: given input X, expect output Y. With LLMs, the same prompt can return syntactically correct but semantically different code each time. Testing strategy must evolve.
🔍
Review is non-negotiable
Because the output varies and can be plausible but wrong, human review is not optional — it's the quality gate. The AI writes the first draft. The engineer is still accountable for the result.
⚡
Creativity becomes a feature
Non-determinism means the model can find solutions humans wouldn't think of first. It can propose 3 different approaches to the same problem, explore design space, suggest alternatives. Variance is a superpower.
📋
Context replaces precision
In assembly, you specify every bit. With LLMs, you specify intent — and the model fills the gaps. The quality of output depends on the quality of context you provide, not the exactness of syntax.
🏗️
Architecture stays with humans
LLMs excel at implementing well-defined functions and modules. They struggle with system-level decisions that require understanding business constraints, long-term maintainability, and organizational context.
📈
The skill bar shifts upward
Junior devs with AI can now produce senior-level output for routine tasks. This doesn't eliminate the need for expertise — it raises the floor, and makes true engineering judgment more valuable, not less.

Chapter 01 / 04

What is an LLM?

A Large Language Model is a neural network trained on billions of pages of text. It learned one single trick — predicting the most probable next word. That's it. That's the whole secret.

Imagine hiring someone who has read every book, every article, every Stack Overflow thread ever written — and can recall any of it in milliseconds.
That's your LLM. A world-class autocomplete engine trained on human knowledge.

Scale

Hundreds of billions of parameters

Each parameter is a tiny numerical weight learned during training. Together they encode patterns in human language.

Training

Months of compute on GPUs

Training GPT-4 or Claude cost tens of millions of dollars. You don't train them — you use them via an API.

Output

Tokens, not characters

The model generates one "token" at a time (roughly ¾ of a word). Speed and cost are measured in tokens/second and $/1M tokens.

Chapter 03 / 04

What are AI Skills?

A raw LLM has no instructions, no personality, no specialization. Skills are structured files and techniques that shape how the model behaves — what it knows about your context, what rules it follows, what tools it can use. Think of them as the configuration layer between a generic model and a useful product. This is how Claude, Copilot, and every AI coding tool are built on top of base models.

📋
System Prompt / Instructions
The foundational skill. A hidden text block sent to the model before your message that defines its role, rules and personality. "You are a senior TypeScript developer at a bank. Never suggest unsafe patterns. Always add JSDoc." Every AI product — Claude, Copilot, ChatGPT — runs on a system prompt you never see.
📄
SKILL.md files — Claude's skill system
Claude and other agentic tools can read markdown files from your project before acting. A SKILL.md file contains best practices, conventions, and instructions specific to a task (e.g. "how to create a PPTX", "how to write a REST endpoint in our stack"). The agent reads it first, then applies it. This is how you encode team knowledge into the AI.
⚙️
.github/copilot-instructions.md
GitHub Copilot reads a special instructions file at the root of your repo. Put your team's coding conventions there: "We use async/await only. No callbacks. All errors must be typed. Tests use Vitest." Every developer on the team gets consistent AI behavior tuned to your codebase — automatically, with no extra prompting.
🔧
Tool Use / Function Calling
The model can invoke external tools — run a terminal command, query a database, call an API, read a file. This transforms the model from a text generator into an autonomous actor. Claude Code and OpenCode use this to read your project files and execute code edits directly.
🎯
Fine-tuning
Retraining a base model on your specific data so it internalizes your patterns. GitHub Copilot was fine-tuned on millions of public code repos — it "thinks" in code by default. Fine-tuning is expensive and reserved for large-scale, stable use cases.
📚
RAG — Retrieval Augmented Generation
Before the model responds, relevant documents are fetched from your private knowledge base and injected into the context. The model reasons over your internal docs, not just its training data. Used to give AI access to private codebases, wikis, runbooks and architecture docs without retraining.
🌡️
Temperature
Controls creativity vs. determinism. Temperature=0 → always picks the most likely token (great for code). Temperature=1 → more varied, creative output. Coding tools use low temperatures.
Think of the base LLM as a raw diamond. Skills are the cutting and polishing. The same diamond can become a ring, a lens, or a drill bit — depending on how it's shaped.
Every AI coding tool is a shaped version of the same underlying technology.

What is MCP?

Model Context Protocol is an open standard that lets AI models connect to any external tool, data source or service — through a unified interface. Think of it as USB-C for AI: one protocol, infinite integrations.

Before MCP, each AI tool had its own way to connect to databases, APIs, and file systems. MCP standardises everything — one protocol that any model can speak, any tool can implement.
Announced by Anthropic in November 2024. Now adopted by OpenAI, Google DeepMind, and most major AI tooling vendors.
What MCP enables concretely

Database access
Claude Code queries your Postgres or MongoDB directly, with schema awareness.
🌐
Web search & browsing

Internal tools & APIs
Connect to Jira, Confluence, Slack, your CI pipeline — all via MCP servers you control.

Read, write and navigate your entire codebase — not just the open file, the whole project.

Chapter 05 / 05

What is an AI Agent?

An assistant answers. An agent acts. An agent is an LLM that can observe its environment, make decisions, use tools, and loop — autonomously — until the task is done. This is the shift from "AI helper" to "AI colleague".

Copilot is a GPS that tells you where to turn. Claude Code is a driver who takes you to the destination — while you sleep.
The fundamental difference between autocomplete and agentic AI.
1
🎯 Observe
The agent reads files, runs commands, checks test output — it understands the current state of your codebase.
2
🧠 Plan
It decides what to do next: which files to edit, which tests to run, which APIs to call — all reasoning done internally.
3
⚡ Act
It executes: edits files, runs commands, calls APIs, creates new files. Real changes in your real codebase.
4
✅ Verify
It checks results: did the tests pass? Is the build green? If not, it loops back to step 1 and self-corrects.
↩ Loops autonomously until the task is done — or asks for human input when blocked

GitHub Copilot
Autocomplete + Agent
Microsoft/GitHub's flagship. Inline autocomplete in every major IDE. Now also has a chat interface and agentic "Workspace" mode for PRs and issues.

🤖
Claude Code
Terminal Agent
Anthropic's terminal-native coding agent. Reads your entire codebase, reasons over it, and autonomously edits multiple files. Best-in-class for complex tasks.

🔓
OpenCode
Open Source Agent
MIT-licensed terminal agent. Identical agentic loop to Claude Code but provider-agnostic — connects to any of 75+ LLM providers, including local models. Zero code storage.

Cursor
AI-Native IDE
A full VS Code fork where AI is built into the editor itself, not a plugin. Has its own "Composer" mode that edits multiple files simultaneously from a chat interface inside the IDE.
How: VS Code fork → embeds GPT-4/Claude directly → Composer = multi-file edits from IDE chat
♊
Gemini Code Assist
Autocomplete + Chat
Google's answer to Copilot. Powered by Gemini models. Deep integration with Google Cloud, BigQuery, and Workspace. Strong for enterprises already on GCP infrastructure.

Codex / ChatGPT for coding
API + Chat Agent

🌊
Windsurf (Codeium)
AI-Native IDE

Tabnine
Privacy-first Autocomplete
Focused on enterprises with strict privacy requirements. Can run its models entirely on-premise. Trains on your private codebase to learn your team's specific patterns and conventions.
How: Fine-tuned smaller model → runs locally or on-premise → learns your codebase patterns → IDE plugin

Open Source CLI Agent
Open-source terminal coding agent with deep Git integration. Every change is automatically committed. Supports multiple LLMs. Loved by developers who want full control and transparency.

Mastering AI Tools — What every dev must know

Having the tool installed is the easy part. The real gap between a dev who gets 20% productivity and one who gets 5x is a handful of techniques. Here's the complete playbook — focused on Copilot since it's your highest-adoption tool, with principles that apply everywhere.

01
The #1 mindset shift
"Stop treating it like Google.

Prompt Engineering — The 5 Rules

Rule 1 — Give a role
❌ "fix this function"
✅ "You are a senior TypeScript developer specializing in security. Review this auth function for vulnerabilities."
A role anchors the model's "perspective". Security dev vs. speed dev give very different reviews.
Rule 2 — Specify the output format
❌ "explain this code"
✅ "Explain this function. Format: 1) what it does, 2) potential issues, 3) suggested improvements as a numbered list."
Without format guidance, you get walls of text. With it, you get structured, actionable output.
Rule 3 — Add context, always
❌ "write tests for this"
✅ "Write Jest unit tests for this Node.js auth service. We use async/await, no callbacks. Cover happy path + 3 edge cases: invalid email, wrong password, missing JWT_SECRET."
Stack, framework, test runner, edge cases — every detail you give saves 3 edit cycles.
Rule 4 — Use "negative prompting"
❌ "refactor this component"
✅ "Refactor this React component for readability. Do NOT change the public API. Do NOT add new dependencies. Do NOT modify the test file."
Telling the model what NOT to do is as powerful as telling it what to do. Saves surprises.
Rule 5 — Think step by step
❌ "migrate this service to async"
✅ "Think step by step. First, identify all synchronous calls. Then list which ones can be made async without breaking the API. Then write the migration plan before changing any code."
"Think step by step" literally improves output quality — it triggers chain-of-thought reasoning.

This is non-negotiable and must be a written policy. AI-generated code always goes through code review before merging. No exceptions. Not because AI is bad — because unreviewed code is bad, regardless of origin. AI actually makes this easier: it can review its own output. "Review the code you just wrote for security issues." It will find problems it introduced.

⚠️ Risks to monitor — the other side of the equation
🔐
Security vulnerabilities
AI-generated code can introduce subtle security flaws — SQL injection, insecure JWT handling, missing input sanitization — that look syntactically correct but are logically broken. Studies show ~40% of AI-generated code contains at least one vulnerability when not reviewed. Mandatory security scan on all AI-generated PRs.
🏗️
Maintainability & technical debt
AI generates working code fast — but often verbose, inconsistent with existing patterns, and lacking architectural awareness. Over time, a codebase with heavy AI contribution risks becoming harder to read, harder to evolve, and more fragile. The risk to project longevity is real: code that nobody fully understands is code nobody can safely change. Enforce code style rules and architectural guardrails in your prompts and CI.
⚡
Energy & MegaWatt constraints
Training and running LLMs is extraordinarily energy-intensive. A single GPT-4 query consumes ~10× more energy than a Google search. At scale — millions of daily dev queries — this represents a significant carbon footprint. More critically, data center capacity is already constrained globally: the race for GPU compute is colliding with limits on available MegaWatts. Medium-term, we may face API rate limits, price spikes, or availability windows driven not by software but by physical energy infrastructure. Factor this into your multi-year AI tooling strategy.
